LLM Council Verdict: Skills Architecture

The Question: What is the optimal architecture for managing 70+ diverse skills across multiple disparate projects to minimize token usage while maximizing agent capability? Are there any risks to moving to a project-specific architecture, and how should the transition be executed?
Chairman's Synthesis
Where the Council Agrees: De-Globalize Immediately

Scoping is non-negotiable. The core issue is the global monolithic architecture. Injecting bioinformatics tools into a marketing task exhausts the context window and degrades performance. The heavy specialized tools (especially the `science` plugin) must be immediately removed from the global folder and installed locally at the project level.

Where the Council Clashes: Routing Rules

The original proposal suggested deleting manual routing rules entirely and trusting the agent to pick the right tool natively. The council heavily debated this. The consensus reached is that relying purely on "guessing" will cause hallucinations when an agent has multiple similar tools locally (like `copywriting` and `copy-editing`). Verdict: Keep explicit routing rules, but move them from the global file to local project files.

Blind Spots Caught

The peer reviewers caught that merely deleting the text routing rules in the global config won't actually stop the skills from loading if the plugin folders still live in the global directory. The physical folders must be moved. They also rejected building a "Dynamic Skill Index" as massive over-engineering.

The Recommendation

Adopt a Project-Scoped Hybrid Architecture. Do not trust pure semantic matching. Group your tools by domain, install them locally in their respective projects, and migrate the manual routing rules into the local project instructions (e.g., `GEMINI.md` or `AGENTS.md`).

The One Thing to Do First

Physically cut the science folder from C:\Users\infon\.gemini\config\plugins and paste it into the local plugin directory of whatever project actually requires it.
